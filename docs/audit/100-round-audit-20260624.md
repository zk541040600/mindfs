## Main fixes added by Rounds 51-100

- Added targeted timeout-helper tests for both Pi runtimes.
- Allowed `tel:` Markdown links without widening image protocols.
- Made local CLI token writes atomic and permission-tightened.
- Hardened relay tip URLs and relay JSON response size limits.
- Hardened download URL protocols and plugin output validation.
- Added WebSocket inbound read limit.
- Tightened reused self-signed TLS key permissions.
- Capped self-update artifact downloads and cleaned partial oversized files.
- Restricted credentialed CORS to local/native/MindFS origins.
- Encrypted proof-required raw file responses for E2EE sessions.
- Serialized scheduled task JSON mutations to avoid lost concurrent updates.

## Independent completion-audit remediation

The first completion audit rejected the goal because one confirmed E2EE file-confidentiality issue remained: when E2EE was enabled, file upload content was still sent as plaintext multipart data with only proof headers. This was fixed after the rejection:

- `web/src/services/e2ee.ts` now exposes `encryptEnvelopeBytes`.
- `web/src/services/upload.ts` now sends E2EE uploads as JSON containing encrypted per-file envelopes and `X-MindFS-E2EE: 1`, instead of plaintext multipart bodies.
- `server/internal/api/http.go` now rejects plaintext multipart uploads when E2EE proof is active and decrypts protected upload payloads before saving files.
- `server/internal/api/http_test.go` adds regressions for rejecting E2EE plaintext multipart and decrypting protected upload payloads.

Focused validation after the fix:

```bash

(cd web && npm run typecheck)
```

Both focused checks passed before the full validation was rerun.
